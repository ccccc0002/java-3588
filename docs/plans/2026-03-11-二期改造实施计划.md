# 二期改造实施计划

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 将现有 RK3588 AI 监控项目从“一期基础可运行”推进到“二期真正可用的前后端产品”，并以细粒度阶段交付、可持续验证、多 agent 并行协作为执行原则。

**Architecture:** 采用“双入口产品结构 + 插件化 AI 运行时 + 分阶段可验收交付”的推进方式。首页固定总览驾驶舱与后台管理分层建设；设备、算法、告警、系统管理等模块按独立工作流并行推进；AI 推理调度、抽帧策略、CPU/内存健康作为贯穿全局的技术主线。

**Tech Stack:** Spring Boot, MyBatis, MySQL, ZLMediaKit, RK3588, RKNN, Python plugin runtime, MPP + RGA, front-end templates/static pages, ECharts, SSH-based board verification, TDD/unit/integration/manual verification.

## 一、计划设计原则

1. 阶段划分尽量细，每一阶段都必须有可见成果。
2. 每一阶段都要求“可演示、可验证、可回归”，不接受长时间只改底层没有可见输出。
3. 每一阶段都要有明确的主负责人 agent、并行协作 agent、输入、输出和验收口径。
4. 所有阶段都要满足“前后端真实可用，不交付原型”。
5. 所有阶段都受 CPU / 内存健康、端侧长稳运行约束。
6. 所有高风险能力都采用“TDD + 板端验证 + 页面联调”三层验证。

## 二、多 Agent 协同模型

### Agent 角色池

#### 1. `architect-agent`
负责：总体架构、模块边界、数据模型、跨层约束、并发与资源预算。

#### 2. `planner-agent`
负责：阶段任务拆分、依赖管理、里程碑控制、阶段验收定义。

#### 3. `frontend-agent`
负责：首页驾驶舱、后台管理页、主题、多语言、交互流程、ECharts 集成。

#### 4. `backend-agent`
负责：Spring Boot 接口、数据表/服务层、业务流程、配置写入、权限接入。

#### 5. `ai-runtime-agent`
负责：RKNN 插件体系、算法包标准、推理耗时采集、抽帧策略、并发调度。

#### 6. `media-agent`
负责：ZLMediaKit、MPP + RGA 解码链路、H.264/H.265、实时预览与 OSD。

#### 7. `integration-agent`
负责：前后端联调、跨模块联调、板端实测、首页与后台主流程打通。

#### 8. `qa-agent`
负责：TDD 测试、回归清单、接口验证、页面验收、资源健康验证。

#### 9. `ops-agent`
负责：启动脚本、运行目录、日志目录、授权文件路径、网络配置落地、板端部署。

#### 10. `security-and-audit-agent`
负责：RBAC、操作日志、授权校验、高风险动作确认链路。

### 协同规则

1. 每个阶段明确主导 agent，其他 agent 并行配合，不交叉覆盖同一文件域。
2. 每个 agent 仅负责自己的工作目录与文件域，避免冲突。
3. 每阶段结束时必须沉淀：结果、风险、测试证据、板端验证证据。
4. 任一高风险模块合并前必须经过 `qa-agent` 与 `security-and-audit-agent` 双重检查。

## 五、阶段验收节奏建议

为了避免你长期看不到结果，建议每个阶段都采用以下节奏：

1. 阶段启动前：明确本阶段只交付什么，不扩散。
2. 阶段中期：输出一次可预览版本或板端结果截图。
3. 阶段结束：提供“可演示结果 + 验收清单 + 风险说明”。

建议的节奏：

1. `Phase 1-3`：偏前台可见成果，优先让你尽快看到产品形态变化。
2. `Phase 4-8`：偏后台主流程打通，优先让你验证是不是你真正需要的业务系统。
3. `Phase 9-10`：偏设备并发能力与 AI 核心能力，优先让你判断设备后续容量是否达标。
4. `Phase 11`：整体交付前总验证。

## 七、当前阶段结论

该计划的目标不是把二期拆成形式上的多阶段，而是确保：

1. 你能更频繁看到结果。
2. 你能更早发现偏差。
3. 多 agent 并行时不会互相踩踏。
4. 高风险能力（并发、抽帧、授权、RBAC）不会被拖到最后。
5. 二期最终交付不是一堆模块，而是一套真正可用、资源健康、可长期运行的系统。
