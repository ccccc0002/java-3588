# 二期改造 PRD 草案

- 日期：2026-03-11
- 状态：草案
- 约束：未收到用户“开始”指令前，仅进行需求整理、PRD 设计、范围确认，不进入实现。
- 来源：基于当前会话中已确认的一期遗留问题、二期新增需求与用户逐条确认结果整理。

---

## 1. 项目背景

当前项目已在 RK3588 平台完成一期基础链路打通，具备如下能力：

1. Java 主应用可启动。
2. ZLMediaKit 已接入运行。
3. RTSP 实时流基础链路已可用。
4. RKNN Python 插件化 yolov8n 已能跑真实流。
5. 告警图片已从原图修正为 AI 标注图。
6. `frame.source` 缺失误用测试图的问题已修复。
7. RKNN 后处理错误解析导致的异常分数/零框问题已修复。

一期已验证链路：

RTSP 真流 -> RK3588 解码 -> RKNN 推理 -> 告警入库 -> 标注图预览

但当前系统整体仍未达到“完整产品可交付”状态，主要问题包括：

1. 大量前后端功能未完全联调可用。
2. 首页与后台信息架构不符合 AI 监控产品形态。
3. 设备、算法、告警、统计、系统管理等模块边界尚未收束。
4. 并发能力、资源健康与长期稳定运行尚未形成系统性设计。

因此启动二期改造。

---

## 2. 二期目标

二期目标不是继续做局部修补，而是完成“端侧真正可用的前后端 AI 平台”改造。

二期总目标：

1. 将系统重构为“固定总览驾驶舱首页 + 后台管理”的双入口产品形态。
2. 完成后台 8 个核心业务模块的真正可用化。
3. 强化 RK3588 平台上的插件化 AI 推理体系与并发能力。
4. 在保证 CPU / 内存健康的前提下实现长期稳定运行。
5. 建立二期正式交付基线，支持后续继续扩展。

---

## 3. 二期交付原则

### 3.1 真正可用，不做原型交付

二期交付的是“真正可用的前后端系统能力”，不是原型、静态页面或假数据演示。所有功能必须满足：

1. 前后端真实打通。
2. 数据真实可读写。
3. 操作结果真实生效。
4. 页面、接口、数据库、运行时链路形成闭环。

### 3.2 CPU / 内存健康是全局硬约束

整个二期改造过程中和完成后，都必须充分保证 CPU 与内存健康，确保 RK3588 长稳运行。该要求作用于：

1. 首页驾驶舱渲染。
2. 后台管理页面。
3. 流媒体与解码链路。
4. AI 推理与调度。
5. 告警、统计、推送等辅助能力。

### 3.3 端侧长期稳定运行优先

二期所有设计与实现，都必须优先考虑：

1. 设备长期在线稳定性。
2. 资源使用可控性。
3. 并发任务可扩展性。
4. 故障后的可恢复性。

---

## 4. 二期技术前提与硬约束

以下约束已确认，不可变更：

1. 硬件平台必须是 RK3588。
2. 流媒体服务器必须使用 ZLMediaKit。
3. 解码链路必须使用 MPP + RGA。
4. 必须支持 H.264 和 H.265。
5. AI 推理必须采用插件化架构。
6. 每个算法是独立算法包。
7. 算法插件优先使用 Python 实现，并参考 Rockchip 官方生态。
8. 当前默认优先跑通的算法为 yolov8n。
9. 日志、产物、告警图等运行数据统一落 SSD，不放在 eMMC 根目录。

## 5. 产品信息架构

### 5.1 双入口结构

#### 入口一：首页固定总览驾驶舱

4. 平台核心状态摘要。

#### 入口二：后台管理

后台管理是正式业务入口，承担全量业务配置与深度管理。

一级模块共 8 个：

1. 总览首页
2. 设备管理
3. 算法管理
4. 告警管理
5. 训练管理
6. 统计管理
7. 推送管理
8. 系统管理

## 6. 首页固定总览驾驶舱 PRD

### 6.1 首页定位

首页是固定总览驾驶舱，不是深层运营调度页，不是后台配置页。

核心设计原则：

1. 固定布局。
2. 强展示、弱操作。
3. 高信息密度但分区清晰。
4. 符合 AI 产品与智能监控驾驶舱风格。

### 6.2 首页信息结构

首页建议固定包含以下区域：

1. 顶部：平台标题、时间、系统状态摘要。
2. 左侧：核心指标卡片。
3. 中央：多宫格实时视频预览区。
4. 右侧：最新告警列表 + 告警详情抽屉/弹层。
5. 底部：统计报表区。

### 6.3 左侧指标卡片

固定展示 4 项：

1. 今日告警数
2. 在线摄像头数
3. 算法数
4. 模型数

### 6.4 中央视频预览区

要求：

1. 首页视频流预览前置。
2. 支持 1 / 4 / 9 / 16 宫格切换。
3. 默认 1 分屏。
4. 不做系统自动主画面选择。
5. 用户自行选择播放哪一路视频。
6. 视频区支持 OSD 实时叠加。
7. OSD 与算法结果、告警图片保持一致。

### 6.5 右侧告警区

要求：

1. 展示最新告警列表。
2. 列表按区域高度自适应滚动显示。
3. 点击告警项后，打开告警详情抽屉或弹层。
4. 不跳离首页主视图。

详情中至少展示：

1. 告警标注图。
2. 告警时间。
3. 摄像头名称。
4. 算法名称。
5. 告警类别 / 中文类别。
6. 置信度。
7. 基础 bbox 摘要。
8. 跳转后台完整告警页入口。

### 6.6 底部统计报表区

固定同时展示三块：

1. 趋势图。
2. 分类饼图。
3. 排名柱状图。

当前阶段统计口径优先围绕告警数据：

1. 告警走势。
2. 告警类型占比。
3. 摄像头或算法 Top N 排名。

### 6.7 首页主题与风格

首页需纳入统一主题体系，支持 4 套主题：

1. 亮色主题
2. 暗色主题
3. 科技蓝驾驶舱主题
4. 苹果简约风主题

## 7. 后台管理 PRD

### 7.1 总览首页

定位：后台内部业务总览页，与外部首页驾驶舱区分。

职责：

1. 业务概览。
2. 快速入口。
3. 管理视角状态摘要。

### 7.2 设备管理

#### 7.2.1 区域管理

1. 支持任意多级树形结构。
2. 不写死园区 / 楼栋 / 楼层 / 房间层级。
3. 区域节点支持父子关系。

#### 7.2.2 摄像头管理

1. 摄像头与数据源在同一页面统一管理。
2. 摄像头直接绑定一个数据源记录。
3. 数据源支持 ONVIF / RTSP。
4. 摄像头新增默认空态。
5. 用户自行选择：扫描 ONVIF 或手动新增。
6. 系统提供入口，不做全自动默认策略。
7. 摄像头列表按区域树组织展示。
8. 支持一键展开。

#### 7.2.3 摄像头算法配置

1. 一个摄像头可绑定多个算法。
2. 在摄像头详情中展示“已绑定算法列表”。
3. 用户逐个进入某个算法的配置上下文。
4. 不采用多个算法在一个页面完全平铺同时配置。

配置内容：

1. 默认阈值读取自算法包。
2. 用户可在页面上覆盖调整阈值。
3. 阈值覆盖值属于“摄像头 x 算法”的绑定配置。
4. ROI 配置在此完成。

#### 7.2.4 ROI 配置

1. ROI 不作为独立页面。
2. ROI 是摄像头-算法绑定关系下的配置项。
3. 基于当前摄像头实时画面抓拍进行绘制。
4. 支持绘制多个检测区域。
5. 多 ROI 必须真实进入后处理逻辑生效。

### 7.3 算法管理

#### 7.3.1 算法包导入

1. 通过后台页面上传一个标准算法包压缩文件。
2. 系统进行结构校验。
3. 校验通过后完成解包、注册与入库。
4. 校验失败必须阻止脏数据进入系统。

#### 7.3.2 算法包列表

1. 当前阶段纯列表展示。
2. 不按类型、运行时、状态分组。

#### 7.3.3 算法包编辑

后台仅允许在线编辑：

1. 名称
2. 描述
3. 标签映射

后台不允许在线编辑：

1. 默认阈值
2. inference/postprocess 源码
3. 模型文件
4. 其他运行时实现逻辑

#### 7.3.4 算法包删除

1. 支持强制删除。
2. 删除前必须先解除绑定关系。
3. 删除时必须二次确认。
4. 删除过程需进入操作日志与权限控制链路。

#### 7.3.5 算法包级配置边界

1. `alert_label` 属于算法包配置。
2. 标签映射属于算法包配置。
3. 摄像头侧不负责重新定义算法包默认告警类别规则。

#### 7.3.6 模型测试

输入方式同时支持：

1. 上传图片测试
2. 选择已存在摄像头抓拍一帧测试

结果页最小展示内容：

1. 原图
2. 标注图
3. 检测结果列表

### 7.4 告警管理

1. 所有功能都要真实可用。
2. 支持多条件查询。
3. 查询条件至少包括：
   - 摄像头
   - 算法
   - 告警类型
   - 时间
4. 时间筛选支持：
   - 预设快捷时间
   - 自定义时间范围
5. 告警详情与标注图完整联动。

### 7.5 训练管理

当前阶段优先目标：训练样本标记与状态流转管理。

要求：

1. 样本记录管理。
2. 标记状态管理。
3. 状态流转管理。
4. 操作人员记录。
5. 操作时间记录。
6. 过程可追踪、可审计。

当前阶段不纳入核心范围：

1. 真正训练任务发起。
2. 训练调度与编排。
3. 完整 MLOps 闭环。

### 7.6 统计管理

定位：深度统计分析页，不是固定静态看板。

要求：

1. 支持用户自由筛选组合出统计结果。
2. 当前至少支持以下筛选维度：
   - 时间
   - 摄像头
   - 算法
3. 图表以折线图、柱状图、饼图为主。
4. 与首页固定驾驶舱统计形成“总览 + 深度分析”分层。

### 7.7 推送管理

“短信管理”统一更名为“告警推送”。

子模块：

1. HTTP 推送
2. 短信推送
3. 电话推送

#### 7.7.1 HTTP 推送

1. 支持配置多个 HTTP 推送目标。
2. 每个目标独立管理。
3. 每个目标独立配置推送地址。
4. 每个目标支持 Bearer 鉴权。
5. 支持断点续传 / 补偿相关能力。
6. 支持自定义上传鉴权文件。

### 7.8 系统管理

二级结构：

1. 授权管理
2. 网络管理
3. 用户与角色管理
4. 操作日志
5. 主题与语言
6. 品牌展示配置

#### 7.8.1 授权管理

当前阶段只做两项核心授权维度：

1. 路数授权
2. 失效时间授权

要求：

1. 基于设备唯一 ID 生成授权 key。
2. 由外部鉴权工具生成 license 文件。
3. license 约束路数与有效期。
4. 达到路数上限时限制继续新增摄像头/数据源。

#### 7.8.2 网络管理

当前阶段只做基础网络管理：

1. 查看双网口信息。
2. 配置 IP。
3. 配置网关。
4. 配置 DNS。
5. 展示每个网口 MAC 地址。

#### 7.8.3 用户与角色管理

当前默认角色类型：

1. 超级管理员
2. 运维人员
3. 只读用户

RBAC 粒度：

1. 至少细到页面按钮 / 操作动作级别。
2. 不只做菜单级访问控制。

只读用户边界：

1. 可以看首页驾驶舱。
2. 可以看后台统计分析。
3. 可以看告警详情。
4. 可以看设备详情。
5. 不允许任何写操作。

#### 7.8.4 操作日志

当前阶段只记录关键写操作，不记录纯查看类行为。

重点覆盖：

1. 新增
2. 编辑
3. 删除
4. 导入
5. 绑定 / 解绑
6. 配置变更
7. 授权变更
8. 推送目标变更
9. 用户与权限变更

#### 7.8.5 主题与语言

主题：

1. 亮色
2. 暗色
3. 科技蓝驾驶舱
4. 苹果简约风

语言：

1. 中文简体
2. 中文繁体
3. 英文

语言入口：登录后右上角全局入口。

#### 7.8.6 品牌展示配置

支持：

1. 软件标题可配置。
2. LOGO 可配置。
3. 登录页背景图可配置。

## 8. AI 插件与运行时 PRD

### 8.1 插件体系原则

1. 每个算法都是独立算法包。
2. 算法包至少包含：
   - 模型文件
   - 配置文件
   - inference 代码
   - postprocess 代码
   - 标签映射
3. `alert_label` 属于算法包自身配置。
4. 算法包是正式交付物，不在后台在线修改核心实现。

### 8.2 推理耗时字段

算法包需要支持推理时间相关字段，以供：

1. 并发评估。
2. 任务调度。
3. 自动抽帧策略。

### 8.3 二期核心优先级：并发能力设计

以下两项已提升为二期核心优先级，不视为可选优化：

1. RK3588 三核推理资源利用优化。
2. 基于算法推理时间与并发任务数的自动抽帧策略。

### 8.4 RK3588 三核推理优化方向

1. 三个 NPU core 的合理利用。
2. 多线程或多进程并行策略。
3. 多算法、多摄像头并发调度模型。
4. 在吞吐与稳定性之间平衡。

### 8.5 自动抽帧策略方向

1. 基于单算法推理时间。
2. 基于当前并发任务数量。
3. 动态调节抽帧节奏。
4. 在实时性、吞吐、CPU/内存健康之间平衡。

### 8.6 运行时非功能性约束

1. CPU 占用尽可能低。
2. 内存占用尽可能低。
3. 保证长期稳定运行。
4. 支持后续并发规模提升。

## 11. 二期验收标准（草案）

### 11.1 功能可用性验收

1. 首页可展示实时视频、告警、统计、驾驶舱信息。
2. 后台 8 个一级模块入口完整可用。
3. 设备管理主流程可用。
4. 算法包导入/编辑/删除可用。
5. 模型测试双通道可用。
6. 告警管理查询、详情、标注图展示可用。
7. 统计管理筛选分析可用。
8. 推送管理多 HTTP 目标可用。
9. 系统管理关键子模块可用。

### 11.2 前后端打通验收

1. 所有关键页面不是原型页面。
2. 页面操作能触发真实后端逻辑。
3. 配置变更能真实写入并生效。

### 11.3 资源健康验收

1. CPU 使用受控。
2. 内存使用受控。
3. 多任务并发下不出现明显失稳。
4. RK3588 长时间运行具备可接受稳定性。

### 11.4 AI 运行能力验收

1. H.264 / H.265 支持验证。
2. RK3588 三核利用优化方案落地。
3. 自动抽帧策略具备真实生效能力。
4. ROI 在后处理链路中真实生效。

## 12. 二期主要风险

1. ROI 从前端绘制到后处理生效涉及跨层联动，复杂度高。
2. 授权管理涉及设备 ID、license 工具、路数限制，属于底层平台能力。
3. RBAC、操作日志、推送管理都属于平台化基础能力，需要统一建模。
4. AI 推理调度与抽帧自适应直接关系设备并发能力，是架构级风险点。
5. 多主题、多语言若不先纳入组件层设计，会导致后续大规模返工。
6. 首页大屏化设计若实现过重，会直接影响 CPU / 内存健康。
