# PRD与架构协同总计划（8-12周，多阶段并行开放）

## 摘要
本计划基于现有工程与既定决策，构建“媒体能力开放 + AI能力开放 + 平台治理开放”的并行演进路线。  
已锁定前提：
1. 流媒体服务器采用 ZLMediaKit（外部独立部署）。
2. 实时预览采用 HTTP-FLV（继续 flv.js）。
3. RK3588 侧推理链路采用 RKNN Runtime，解码/图像处理采用 FFmpeg + MPP + RGA，本地推理微服务与 Java 平台通过 HTTP 协同。
4. 协作模式采用按域并行小队。
5. 开放范围采用 API 与插件同时开放。

## 一、产品目标与成功标准（PRD）

### 1. 业务目标
1. 在 8-12 周内完成“可上线的 ZLM + RK3588 推理协同平台”。
2. 实现多团队并行开发，不以单主线串行阻塞交付。
3. 实现对外开放能力，支持后续第三方算法/设备快速接入。

### 2. 成功指标（验收 KPI）
1. 实时预览端到端延迟：P95 小于 1200ms（HTTP-FLV）。
2. 推理结果上报成功率：大于等于 99.5%。
3. 流可用性：单路 24h 稳定拉流成功率大于等于 99.9%。
4. 告警链路完整率：大于等于 99.0%（检测 -> 上报 -> 展示 -> 通知）。
5. 并行交付效率：四个域小队每周均可独立合并至少 1 个可验收增量。
6. 回滚时间：配置级回滚小于 10 分钟。

### 3. 非目标（当前阶段不做）
1. 不做 WebRTC 全量替代。
2. 不做 Java 进程内嵌 ZLM。
3. 不做跨机房复杂调度平台。
4. 不做训练平台与模型标注全链条重构。

## 二、用户与场景

### 1. 角色
1. 平台管理员：配置媒体、设备、算法与告警策略。
2. 算法工程师：发布/调试 RK3588 推理插件。
3. 运维工程师：监控 ZLM、推理服务与平台健康。
4. 业务查看者：实时预览、告警审阅和统计分析。

### 2. 关键场景
1. 摄像头上线后自动获得推流地址并可预览。
2. 算法服务按配置拉取任务，推理后回传结构化结果。
3. 告警在页面 OSD 与告警列表同步展示。
4. 出现异常时可快速降级到 legacy 流链路。

## 三、架构师 Agent 设计输出（用于协同开发）

### 1. 架构原则
1. 契约优先：先定 API/事件/错误码，再并行实现。
2. 可切换：`legacy` 与 `zlm` 双模式共存，配置开关控制。
3. 解耦：媒体层、推理层、平台层、前端层通过稳定接口连接。
4. 可观测：每条链路都有 trace id、指标、审计日志。

### 2. 目标架构（逻辑分层）
1. 媒体域：ZLMediaKit + Stream URL Resolver + 流状态管理。
2. AI域：RK3588 本地推理微服务（RKNN + MPP/RGA）+ 模型插件加载。
3. 平台域：Java 管理端（配置、设备、告警、权限、任务编排）。
4. 交互域：前端播放（flv.js）+ websocket OSD + 运营页面。
5. 治理域：鉴权、限流、审计、监控告警、发布控制。

### 3. 并行小队职责（RACI 简化）
1. 媒体小队：ZLM 接入、播放与推流地址规则、链路稳定性。
2. AI小队：RK3588 推理服务、插件接口、模型生命周期。
3. 平台后端小队：配置中心、控制器改造、任务与状态编排。
4. 前端小队：播放改造、OSD 持续兼容、状态可视化。
5. 架构师 Agent：接口评审、跨队依赖拆解、风险把关、发布门控。

## 四、开放接口与类型变更（决策完成）

### 1. 配置项新增（`tbl_biz_config`）
1. `media_server_type`：`legacy|zlm`。
2. `zlm_enable`：`0|1`。
3. `zlm_schema`：`http`。
4. `zlm_host_public`、`zlm_host_inner`。
5. `zlm_http_port`、`zlm_rtmp_port`、`zlm_app`。
6. `zlm_play_mode`：`stream|proxy`。
7. `infer_backend_type`：`legacy|rk3588_rknn`。
8. `infer_service_url`：本地推理微服务地址。
9. `infer_timeout_ms`、`infer_retry_count`。

### 2. 后端接口与类型新增
1. 新增 `StreamUrlResolver` 接口与 `LegacyResolver`、`ZlmResolver` 两套实现。
2. 新增 `InferenceClient` 接口与 `Rk3588InferenceClient` 实现。
3. 新增插件契约：
   - `PluginManifest`：`plugin_id`、`version`、`runtime`、`capabilities`。
   - `InferenceRequest`：`camera_id`、`frame_meta`、`model_id`、`roi`。
   - `InferenceResult`：`detections`、`latency_ms`、`trace_id`。

### 3. 现有接口行为调整
1. `/api/camera/list` 的 `rtmp_url` 在 `zlm` 模式统一为 `rtmp://{inner_host}:{rtmp_port}/{app}/{cameraId}`。
2. `/stream/play_list`、`/stream/camera_list` 返回完整 `playUrl`，前端不再拼接端口。
3. `start/stop` 在 `zlm` 模式改为流状态编排，不依赖端口抢占。
4. 保留 websocket `/report/{uid}` 与前端 OSD 渲染协议，字段不破坏性变更。

### 4. 插件开放规范（首版）
1. 插件输入输出 JSON Schema 固定版本号。
2. 插件注册采用“签名包 + manifest + 健康探测”。
3. 插件故障隔离：单插件失败不影响主流程；失败触发熔断和降级策略。

## 五、多阶段并行开放实施计划（8-12周）

### Phase 0（第1周）：契约冻结与基线准备
1. 产出 PRD v1、接口规范 v1、错误码与事件字典 v1。
2. 建立跨队 DoD（Definition of Done）和合并门槛。
3. 完成现网与测试环境基线盘点，定义回滚策略。

### Phase 1（第2-3周）：媒体开放与兼容双轨
1. 媒体小队完成 ZLM 接入、地址解析器、双模式开关。
2. 平台后端小队改造流相关接口走 resolver。
3. 前端小队完成“后端下发完整 playUrl”的播放适配。
4. 交付标准：ZLM 模式下可稳定预览，legacy 可随时切回。

### Phase 2（第3-5周）：AI插件开放与 RK3588 接入
1. AI小队交付本地推理微服务（RKNN + MPP/RGA）最小可用版。
2. 平台后端小队完成 `InferenceClient` 抽象与路由切换。
3. 架构师 Agent 主持插件契约评审与版本策略冻结。
4. 交付标准：至少 1 个算法在 RK3588 链路完成闭环告警。

### Phase 3（第5-8周）：并行联调与可观测治理
1. 建立端到端 trace 贯穿：流创建、推理、上报、展示。
2. 完成告警可靠性策略：重试、去重、幂等、死信处理。
3. 完成灰度策略：按摄像头分组启用新链路。
4. 交付标准：KPI 达标并通过 72h 稳定性测试。

### Phase 4（第8-12周）：灰度发布与规模化协同
1. 分批迁移摄像头到 ZLM + RK3588 链路。
2. 完成运维手册、故障剧本、容量规划与扩容预案。
3. 对外发布“开放能力文档 v1”（API + 插件接入指南）。
4. 交付标准：生产可运行、可回滚、可扩展。

## 六、协同开发机制（并行开放核心）

### 1. 分支与集成策略
1. 按域建立长期分支：`feat/media-zlm`、`feat/ai-rk3588`、`feat/platform-contract`、`feat/ui-preview`。
2. 每周固定两次契约同步，禁止未评审字段变更直入主干。
3. 合并要求：通过单元测试 + 集成测试 + 契约测试。

### 2. Agent 协同流程
1. 架构师 Agent：每周输出“依赖冲突清单 + 风险清单 + 调整建议”。
2. 开发 Agent：按域执行实现计划并产出接口变更记录。
3. 测试 Agent：维护回归矩阵并执行阶段门禁。
4. 评审 Agent：做发布前架构一致性审查与安全审查。

### 3. 决策与升级机制
1. 24h 内无法闭合的跨队冲突升级到架构师 Agent 仲裁。
2. 影响 KPI 的变更必须附带回滚方案和风险评估。

## 七、测试与验收场景

### 1. 功能测试
1. ZLM 模式实时播放正常，页面 OSD 正常叠加。
2. RK3588 推理结果可写入告警并实时推送。
3. 配置切换 `legacy <-> zlm`、`legacy <-> rk3588` 可即时生效。

### 2. 稳定性测试
1. 7x24 小时单路和多路播放稳定性。
2. 推理服务异常重启、网络抖动、上游断流恢复能力。
3. 压测场景下告警漏报率与重复率控制。

### 3. 兼容与回归
1. 旧模板页面与历史接口字段兼容。
2. 旧数据库结构在不迁移情况下可运行。
3. 告警通知链路（企业微信/飞书/API）无回归。

### 4. 安全与治理
1. 插件包签名校验、输入参数校验、鉴权与审计日志。
2. API 限流、错误码规范化、敏感配置脱敏。

## 八、风险与预案

### 1. 关键风险
1. ZLM 与网络环境不一致导致播放失败。
2. RK3588 端模型兼容与性能不稳定。
3. 多队并行导致接口漂移和返工。
4. 插件开放引入安全与稳定性隐患。

### 2. 预案
1. 增加链路探活接口和预检脚本，发布前强制执行。
2. 建立模型白名单与性能基线，未达标禁止灰度。
3. 契约测试前置到 CI，字段漂移直接阻断合并。
4. 插件沙箱与故障隔离，默认最小权限。

## 九、默认假设与已选定参数

1. ZLM 独立部署且可被平台与算法服务访问。
2. 默认 `zlm_app=live`，`zlm_play_mode=stream`。
3. 前端继续采用 flv.js；OSD 继续走 websocket 前端叠加。
4. RK3588 推理服务以 HTTP 本地微服务形态接入。
5. 本期采用“双模式共存 + 灰度迁移 + 快速回滚”作为上线策略。
