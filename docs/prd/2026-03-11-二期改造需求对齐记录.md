## 一、项目背景

当前项目已完成一期基础打通，已经具备 RK3588 端侧基础运行能力，包括：

1. Java 主应用可启动。
2. ZLMediaKit 已接入运行。
3. RTSP 实时流基础链路已可用。
4. RKNN Python 插件化 yolov8n 已能跑真实流。
5. 告警图片已从原图修正为 AI 标注图。
6. `frame.source` 缺失导致误用测试图的问题已修复。
7. RKNN 后处理错误解析导致的异常分数/零框问题已修复。

当前链路已可运行：

RTSP 真流 -> RK3588 解码 -> RKNN 推理 -> 告警入库 -> 标注图预览

## 二、二期改造总目标

二期不再以“能跑通”为目标，而以“完整产品化、可演示、可扩展、可持续演进”为目标，完成以下统一收口：

1. 端侧完整软件链路打通。
2. 前后端完整联调走通。
3. 视频预览、OSD、告警、算法配置形成闭环。
4. RKNN 推理以插件平台方式沉淀，支持持续扩展。
5. 首页与后台管理重新分层，形成更符合 AI 产品风格的交互入口。

## 三、二期不可变技术约束

以下约束视为二期固定前提：

1. 硬件平台必须是 RK3588。
2. 流媒体服务器必须使用 ZLMediaKit。
3. 解码链路必须使用 MPP + RGA。
4. 必须支持 H.264 和 H.265。
5. AI 推理必须采用插件化架构。
6. 每个算法必须是独立算法包。
7. 算法包必须支持：
   - 模型文件
   - 配置文件
   - inference 代码
   - postprocess 代码
   - label / 中文 label 映射
   - alert_label 配置
8. 算法插件优先使用 Python 实现，并参考 Rockchip 官方文档和生态。
9. 当前优先跑通的插件算法为 yolov8n。
10. 日志、产物、告警图等运行数据统一落到 SSD，不放在 eMMC 根目录。
11. 后续开发与验证遵循 TDD 思路推进。

## 四、一期未完成项并入二期范围

### 4.1 前后端联调未完全收口

1. 不是所有页面按钮和配置项都完成了逐项联调验证。
2. 需要逐页验证页面动作是否真正绑定到后端接口并生效。
3. 需要补齐交互状态、异常提示、空态、加载态和回写逻辑。

### 4.2 实时预览与 OSD 仍需产品化收口

1. 实时预览虽然已有基础能力，但还未完全达到稳定可展示、可联动、可演示的产品状态。
2. OSD 叠加需要和实时画面、告警事件、算法结果完全统一。
3. 视频流管理页与首页预览能力需要重构。

### 4.3 多算法、多摄像头能力尚未完全成型

1. 摄像头与算法关系需要由单一配置扩展到一对多配置。
2. 不同摄像头后续需要支持挂载多个算法并独立控制。
3. 算法参数、告警策略、告警展示需按算法维度隔离。

### 4.4 H.265 与长稳验证未完成

1. 当前重点验证过 H.264 真流。
2. H.265 需要做真实流验证。
3. 断流重连、长时间运行、分辨率变化、重启恢复等稳定性验证未完成。

### 4.5 部署与运维工程化未收口

1. 一键启动、重启、巡检、回滚和日志治理仍需补强。
2. 中间件、前端、后端、流媒体、推理运行时需要统一运维视角。
3. 板端异常恢复机制需要标准化。

## 五、已确认的核心功能需求

### 5.1 流媒体与实时预览

1. 首页必须能直接看到视频实时预览。
2. 视频流管理中的实时预览必须可用，不能只是静态占位。
3. 预览链路由 ZLMediaKit 承担流服务能力。
4. 预览画面需要支持 OSD 叠加，并与算法结果同步。
5. 后续需要支持多分屏布局和设备切换。

### 5.2 AI 插件化算法平台

1. RKNN 推理必须做成插件模式。
2. 每个算法独立成包，便于后续自定义扩展。
3. 插件配置文件必须可编辑，支持：
   - 检测类别
   - 置信度阈值
   - NMS 阈值
   - label 到中文 label 映射
   - alert_label
4. alert_label 的规则为：只有命中 alert_label 的类别才告警。
5. 当前先以 yolov8n 为默认算法跑通。

### 5.3 告警体系

1. 告警图片必须是 AI 标注图，而不是原图。
2. 告警记录需要支持图片预览。
3. 告警详情和历史数据需要和告警图片、告警类别、摄像头、算法关联展示。
4. 告警事件、OSD 展示、告警图片三者要一致。

1. 视频流管理预览要前置到首页，而不是深藏在后台页面里。
2. 首页作为“AI 实时监控总览入口”，优先展示实时画面。
3. 首页可参考用户提供的示意图，包含但不限于以下元素：
   - 顶部大标题和全局时间信息
   - 中央多宫格视频预览区
   - 左侧设备目录 / 学校 / 楼栋 / 教室树形导航

   - 右侧当日告警与告警事件列表
   - 底部或侧边告警图像区域
4. 首页的主入口偏“总览大屏 + 运营驾驶舱”定位。

### 6.3 增加统计报表与可视化分析

1. 增加柱状图、饼状图、趋势图等可视化统计报表。
2. 报表需要至少覆盖：
   - 告警走势
   - 各类型告警分布
   - 摄像头告警排名

3. 可使用 ECharts 等更美观、更有科技感的可视化插件。
4. 图表设计要兼顾“好看”和“可读”，不是简单堆组件。

### 6.4 首页与后台管理重新分层

1. 首页承担可视化总览、视频预览、告警态势展示职责。
2. 其他细项功能收纳到后台管理。
3. 后台管理作为正式业务入口，需要承载：
   - 摄像头配置
   - 算法管理
   - 参数配置
   - 告警管理
   - 报表查看
   - 系统配置
4. 首页与后台管理的职责边界需要明确，避免重复设计。

## 十二、2026-03-11 新增确认结论

### 12.1 首页角色定位已确认

用户已明确确认：首页定位为“监控大屏展示”，而不是“展示+复杂配置操作”的混合模式。

据此，二期信息架构进一步收敛为：

1. 首页：监控大屏 / 态势总览 / 实时预览主界面
2. 后台管理：业务配置、参数维护、算法管理、摄像头管理、告警管理等操作入口

### 12.2 首页职责边界

首页主要承担以下职责：

1. 多宫格实时视频预览
2. 设备树切换与查看
3. OSD 实时叠加展示
4. 告警事件摘要与告警图像展示
5. 指标卡片与趋势图表展示
6. AI 平台总览风格展示

首页不承担以下职责：

1. 复杂表单配置
2. 摄像头参数维护
3. 算法插件安装或编辑
4. 详细告警规则配置
5. 系统管理操作

上述操作统一进入后台管理。

### 12.3 对二期设计的直接影响

基于该确认，二期首页设计应偏向“大屏化、弱操作、强可视化”，具体原则为：

1. 首页优先级最高的是画面、告警、态势、统计。
2. 首页交互以切换查看、筛选、钻取为主，不以编辑配置为主。
3. 后台管理通过独立入口进入，保持管理页面的业务完整性。
4. 首页视觉风格可以更大胆、更偏智能监控/AI 态势驾驶舱，而不必受传统后台表格布局限制。

## 十三、2026-03-11 新增确认结论（二）

### 13.1 首页展示形态已确认

用户已明确确认：首页更偏“固定总览驾驶舱”，而不是“按学校/园区/楼栋/设备树持续切换的运营型监控大屏”。

### 13.2 设计含义

据此，首页在二期中的定位进一步明确为：

1. 固定布局的总览驾驶舱。
2. 重点突出平台总体态势，而不是深层级业务导航。
3. 首页的核心是“总览展示”，不是“以设备树驱动的多层运营调度页面”。

### 13.3 首页建议信息结构

基于当前确认，首页建议优先固定展示以下区域：

1. 顶部：平台标题、时间、系统运行状态摘要。
2. 左侧：核心指标卡片（告警数、摄像头数、算法数、模型数等）。
3. 中央：多宫格实时视频预览区。
4. 右侧：当日告警摘要、最新告警事件。
5. 下方：告警趋势、分类占比、告警图像等统计与辅助信息。

### 13.4 对设备树的影响

1. 设备树不再作为首页主驱动结构。
2. 若保留设备树，也只作为辅助信息或轻量筛选入口。
3. 深层级设备管理与配置操作统一放到后台管理中完成。

### 13.5 对二期 UI 重构的影响

首页视觉与交互原则进一步收敛为：

1. 固定总览布局优先。
2. 强展示、弱操作。
3. 高信息密度但分区清晰。
4. 视觉风格偏 AI 驾驶舱、智能监控总览，不偏传统后台列表页。

### 14.1 首页视频宫格能力已确认

用户已明确确认：首页实时视频区需要支持以下宫格切换能力：

1. 1 分屏
2. 4 分屏
3. 9 分屏

### 14.2 默认宫格已确认

用户表述为“默认是1分配”，当前按“默认 1 分屏”理解并记录。

即：首页首次进入时，默认展示 1 分屏主画面；用户可在页面上切换为 4 / 9 / 16 分屏。

### 14.3 对首页交互的影响

据此，首页视频区域需要满足以下设计要求：

1. 宫格切换控件需固定可见，位置明确。
2. 默认进入时优先展示单路主画面。
3. 切换宫格时，布局切换应平滑、直观。
4. 不同宫格模式下，视频卡片都需要保持统一的标题、状态、OSD 展示逻辑。
5. 宫格切换属于首页查看交互，不涉及后台配置页。

### 14.4 首页视频区建议行为

基于当前已确认需求，首页视频区建议行为如下：

1. 默认 1 分屏显示重点画面。
2. 提供 1 / 4 / 9 / 16 分屏快速切换按钮。
3. 在 1 分屏模式下优先突出主画面与告警态势。
4. 在多分屏模式下突出总体监控覆盖面。
5. 后续若要支持轮播、自动切屏、重点通道置顶，可作为二期增强项或三期扩展项评估。

## 十五、2026-03-11 新增确认结论（四）

### 15.1 首页主画面选择策略已确认

据此，首页视频区交互继续收敛为：

1. 用户自主选择要播放的摄像头通道。
2. 用户再根据需要切换 1 / 4 / 9 / 16 分屏。
3. 系统不主动做主画面智能编排。
4. 若未来需要“告警优先主画面”或“自动轮播”机制，可作为后续增强功能单独规划，不纳入当前二期基础范围。

## 十六、2026-03-11 新增确认结论（五）

### 16.1 首页告警事件区展示方式已确认

用户已明确确认：首页右侧告警事件区需要采用以下展示模式：

1. 展示最新告警列表。
2. 支持点击告警后，打开告警详情抽屉或弹层预览。

### 16.2 当前明确交互策略

据此，首页告警区域不只停留在静态列表，而需要形成“摘要 + 快速预览”的交互闭环：

1. 首页右侧先展示最新告警列表。
2. 用户点击某条告警后，在当前首页上下文中打开详情抽屉或弹层。
3. 详情中需支持查看告警图、告警时间、摄像头、算法、类别、置信度等核心信息。
4. 用户无需离开首页即可完成快速查看。

### 16.3 对首页定位的影响

该设计仍然符合首页“监控大屏展示”的定位，因为：

1. 它增强的是信息查看能力，而不是后台配置能力。
2. 抽屉或弹层属于轻交互，不会把首页变成复杂管理页。
3. 告警详情预览有助于在大屏总览场景中快速确认事件。

### 16.4 建议的详情内容范围

首页告警详情抽屉/弹层建议至少展示：

1. 告警标注图。
2. 告警时间。
3. 摄像头名称。
4. 算法名称。
5. 告警类别 / 中文类别。
6. 置信度。
7. 基础位置信息或 bbox 摘要。
8. 跳转到后台完整告警管理页的入口。

## 二十五、2026-03-11 新增确认结论（十）

### 25.1 ROI 配置归属已纠正

用户已明确确认：ROI 配置不应作为设备管理中的独立页签单独存在。

ROI 的正确归属方式为：

1. 在摄像头或数据源配置算法时进行配置。
2. ROI 属于“摄像头/数据源 与 算法绑定关系”的配置内容。
3. ROI 不是设备本身的基础属性，而是算法执行范围的一部分。

### 25.2 对信息架构的直接影响

据此，设备管理与算法管理之间的边界进一步明确：

#### 设备管理负责

1. 区域管理
2. 摄像头管理
3. 数据源管理
4. 摄像头基础信息维护
5. 摄像头/数据源与算法的绑定入口

#### 算法绑定配置负责

1. 选择摄像头挂载哪些算法
2. 配置算法参数
3. 配置 alert_label 等规则
4. 配置 ROI 检测区域
5. 保证 ROI 配置最终进入后处理执行逻辑

### 25.3 当前推荐的设备管理二级结构修正

基于该确认，设备管理二级结构不再理解为：

1. 区域管理
2. 摄像头管理
3. 数据源管理
4. ROI 配置

而修正为更合理的结构：

1. 区域管理
2. 摄像头管理
3. 数据源管理
4. 摄像头算法配置（其中包含 ROI 配置）

### 25.4 设计原则

1. ROI 必须跟随具体算法配置生效，而不是孤立保存。
2. 同一摄像头挂不同算法时，可存在不同 ROI 配置。
3. ROI 保存后，后处理逻辑必须真实使用该 ROI，而不是仅做前端展示或数据库留痕。

### 25.5 当前阶段结论

后续若继续细化设备管理模块，应按以下方向理解：

1. 设备管理负责对象管理。
2. 摄像头算法配置负责运行规则管理。

1. 纯列表结构。
2. 优先保证导入、查看、编辑、删除等主流程清晰。
3. 不在当前阶段引入额外分组复杂度。

若后续规模增长明显，可再评估增加分组、标签或筛选能力。

## 五十一、2026-03-11 新增确认结论（三十六）

### 51.1 二期交付标准已进一步明确

用户特别强调：二期要求交付的是“真正可用的前后端系统能力”，而不是原型设计或半成品页面。

### 51.2 当前明确的实现标准

据此，二期中所有前后端功能均应按以下标准理解：

1. 不是静态原型页。
2. 不是仅完成页面样式展示。

### 51.3 当前明确的全局非功能性约束

用户同时特别强调：

1. 实现过程中要充分保证 CPU 健康。
2. 实现过程中要充分保证内存健康。
3. 完成后也要充分保证 CPU 和内存长期稳定。
4. 二期设计与实现都必须以设备长稳运行为前提。

### 51.4 对二期方案的直接影响

该确认意味着二期所有模块都不能只按“功能存在”验收，而需要按“真实可用 + 资源健康”共同验收。具体包括：

1. 首页与后台管理都必须是真实可交互、可联调、可生效。
2. AI 推理优化、抽帧策略、运行时调度必须以资源健康为核心约束。
3. 前端设计不能引入过重渲染负担。
4. 后端设计不能引入不必要的常驻高占用。
5. 流媒体、推理、告警、统计、管理等功能都要服从端侧长期稳定运行目标。

### 51.5 当前阶段结论

1. 所有前后端功能都要真实可用，不按原型交付。

3. CPU 和内存健康是全局硬约束。

## 五十二、2026-03-11 新增确认结论（三十七）

### 52.1 AI 推理优化与抽帧自适应被提升为二期核心优先级

用户特别强调，以下两项不是普通优化项，而是关系到设备后续并发规模与可交付能力的核心目标：

1. RK3588 三核推理资源利用优化。
2. 基于算法推理时间与并发任务数的自动抽帧策略。

### 52.2 当前明确的业务含义

据此，二期中的 AI 推理链路不能只满足“能跑”，而必须面向“后续并发数量”进行设计。也即：

1. 设备未来能带多少路并发，不只取决于算法精度。
2. 更取决于 RK3588 三个 core 的利用方式。
3. 更取决于多算法、多摄像头并发下的调度策略。
4. 更取决于是否能根据实时负载自动调节抽帧节奏。

### 52.3 当前明确的优化方向

#### 1. RK3588 三核利用优化

需要重点考虑：

1. 三个 NPU core 的合理利用。
2. 多线程或多进程并行调度的可行性。
3. 在不破坏稳定性的前提下提高吞吐。
4. 避免某个 core 长期热点、其他 core 空闲。
5. 与 CPU / 内存健康约束共同权衡。

#### 2. 推理时间字段与抽帧策略自适应

需要重点考虑：

1. 算法包支持“推理时间”相关字段。
2. 系统能够基于单算法推理时间、并发任务数量进行评估。
3. 运行时根据当前负载自动调整抽帧策略。
4. 在吞吐、实时性、稳定性之间做平衡。
5. 抽帧策略不应写死，而应具备动态调节能力。

### 52.4 对二期架构设计的直接影响

该确认意味着二期架构设计必须把以下内容视为核心设计输入，而不是后期调优项：

1. 推理调度模型。
2. 多算法并发模型。
3. 摄像头-算法任务队列模型。
4. 推理耗时采集与统计模型。
5. 自适应抽帧控制模型。
6. CPU / NPU / 内存资源预算模型。

### 52.5 当前阶段结论

截至目前，二期中的“AI 推理优化”和“自动抽帧策略”应正式理解为：

1. 直接影响设备并发能力的核心能力。
2. 需要优先纳入架构设计主线。
3. 不是可有可无的后置优化项。
4. 必须与 CPU / 内存健康、长期稳定运行目标一起统筹设计。
